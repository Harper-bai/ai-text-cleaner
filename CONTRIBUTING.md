# Contributing

这个项目目前是一个很小的静态 demo，但它的价值不在页面本身，而在于：

- 真实样本
- 明确规则
- 可重复验证

所以后续贡献请优先围绕这三件事。

## 开始前先读

建议先看这些文件：

- `README.md`
- `ARCHITECTURE.md`
- `DESIGN.md`
- `samples/README.md`

## 本地开发

项目不需要构建。

### 启动一个本地静态服务

```bash
cd ai-word-cleaner-demo
python3 -m http.server 4173
```

然后访问：

```text
http://localhost:4173
```

## 修改规则前的原则

### 1. 先加样本，再改规则

如果你发现某类文本处理得不对：

- 先把它加进 `samples/cases.mjs`
- 明确写出 `expected`
- 再去修改 `src/rules.mjs` 或 `src/cleaner.mjs`

不要先改规则再回头补样本。

### 2. 不要让规则互相污染

这个项目当前只做“正文清洗”。

所以新增规则时要避免：

- 为了一个特殊 case 破坏已有正文样本
- 把“表格、模板、参考文献”这类别的能力硬塞进正文规则

### 3. 保持结果可预测

同一段文本多次运行清洗，结果应该稳定。

也就是：

- 不应该越清洗越乱
- 不应该第一次和第二次结果不同

## 测试方式

每次修改清洗规则后，至少跑一次：

```bash
node ./scripts/run-samples.mjs
```

如果新增样本，也要保证所有旧样本继续通过。

## 什么时候该加新样本

优先加这些真实场景：

- Markdown 标题和列表残留
- 中文空格异常
- 中英混排
- 提纲改正文
- 长段学术文本

暂时不要把这些混进当前版本：

- Markdown 表格
- 参考文献格式
- 学校模板样式
- `.docx` 导出行为

这些属于后续版本的能力，不是当前正文清洗核心。

## UI 修改原则

页面定位是：

- 工具站
- 不是产品发布页
- 不是营销站

所以修改 UI 时：

- 优先保持工具本体可见性
- 少加解释区
- 少加无用按钮
- 少加装饰性层级

所有视觉相关改动都应先对照：

- `DESIGN.md`

## 提交前检查

如果你改了逻辑：

- 跑 `node ./scripts/run-samples.mjs`

如果你改了文档：

- 确认 `README.md`、`ARCHITECTURE.md`、`DESIGN.md` 没有互相打架

## 贡献优先级

目前最有价值的贡献顺序：

1. 新增真实样本
2. 改进清洗规则
3. 改进复制和输入体验
4. 再考虑新功能
